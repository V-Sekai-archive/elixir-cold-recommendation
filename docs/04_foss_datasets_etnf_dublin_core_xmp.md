# FOSS datasets: schema and Dublin Core XMP

Catalog and event schema used for FOSS datasets (e.g., UCI Clickstream). Metadata is exposed as **Dublin Core** [XMP JSON-LD](https://github.com/KhronosGroup/glTF/tree/main/extensions/2.0/Khronos/KHR_xmp_json_ld) generated by the application from `catalog_item` rows (see [Enforcing XMP JSON-LD](#enforcing-xmp-json-ld-rdfex-and-grax) below).

---

## Schema (this repo)

| Relation         | Role              | Key                                                                                                          |
| ---------------- | ----------------- | ------------------------------------------------------------------------------------------------------------ |
| **catalog_item** | One row per item  | `(item_id, source_dataset)`. Columns: `dc_title`, `dc_description`, `dc_type`, `dcterms_source`, timestamps. |
| **event**        | One row per click | `(session_id, ord)`. Columns: `item_id`, `source_dataset`, timestamps.                                       |

---

## Dublin Core

We use the Dublin Core Metadata Element Set and DCMI terms:

| Element                                        | Use                                                          |
| ---------------------------------------------- | ------------------------------------------------------------ |
| **title**                                      | Short description (e.g., first line or first 200 characters). |
| **description**                                | Full item text; use for RecGPT/MPNet and full content.       |
| **identifier**                                 | Same as `item_id` (e.g., `uci_clickstream:42`).               |
| **source**                                     | Provenance: `source_dataset`.                                |
| **type**, **language**, **format**, **rights** | Optional when the dataset provides them.                     |

Namespaces: Dublin Core 1.1 `http://purl.org/dc/elements/1.1/`; DCMI terms `http://purl.org/dc/terms/`.

## XMP JSON-LD

XMP JSON-LD is **generated** from `catalog_item` by the application (see [Enforcing XMP JSON-LD](#enforcing-xmp-json-ld-rdfex-and-grax) below), not by a database view. Each item’s JSON-LD has:

- `@context` mapping `dc` and `dcterms` to DCMI URIs.
- Dublin Core properties for all item metadata.
- XMP-style JSON-LD (simple key–value; no nested `@graph`) so plain JSON and JSON-LD parsers can read it.

**RecGPT:** Use `dc:description` (full text) when building training data from the ETNF DB.

**Example:**

```json
{
  "@context": {
    "dc": "http://purl.org/dc/elements/1.1/",
    "dcterms": "http://purl.org/dc/terms/"
  },
  "dc:title": "category 1 product 42 colour blue",
  "dc:description": "category 1 product 42 colour blue",
  "dc:identifier": "uci_clickstream:42",
  "dcterms:source": "uci_clickstream"
}
```

## Enforcing XMP JSON-LD (RDF.ex and Grax)

Catalog-item XMP JSON-LD is **enforced** via [RDF.ex](https://rdf-elixir.dev/) and [Grax](https://rdf-elixir.dev/grax/): RDBMS → Grax struct → RDF graph → compact JSON-LD with Dublin Core context. Use `RecGPT.Xmp.Jsonld.from_catalog_item/1` to produce schema-conformant XMP JSON-LD from any `catalog_item` row (Ecto struct or map). Use `validate_jsonld/1` to check that existing JSON-LD conforms to the catalog-item schema.

- **RecGPT.Xmp.DublinCore** — DC/DCMI term IRIs and `context/0` for compaction.
- **RecGPT.Xmp.CatalogItemSchema** — Grax schema (Dublin Core properties) for mapping RDF ↔ struct.
- **RecGPT.Xmp.Jsonld** — `from_catalog_item/1`: catalog row (Ecto struct or map) → XMP JSON-LD string. `to_xmp_jsonld_string/2`: RDF data → JSON-LD string with context. `validate_jsonld/1`: parse JSON-LD → RDF → load into schema (validates shape).
